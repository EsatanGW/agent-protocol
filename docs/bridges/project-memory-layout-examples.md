# Project Memory Layout — Concrete Examples

> **Status:** non-normative companion to [`docs/ai-project-memory.md`](../ai-project-memory.md) §Recommended on-disk layout. Nothing in this file is binding; the binding contract is the four runtime requirements (writable, retrievable, versioned, traceable) defined in that section.

This file shows three layouts that satisfy the role split (active manifests, retired manifests, CCKN, completion reports, escalation log, supervision log, index) under different repo shapes. Pick one, mix and match, or design your own — the role split is what matters; the names are for your repo's convention to decide.

---

## Pattern 1 — Single-prefix flat layout

**When to use:** single-repo project, no Git LFS pressure, ≤ a few hundred manifests in flight at any time.

```
.ai-memory/
├── index.md                          # role: index — top-level entry point
├── manifests/
│   ├── active/
│   │   ├── 2026-04-26-search-p95-mission.yaml
│   │   └── 2026-04-22-jwt-rotation.yaml
│   └── retired/
│       └── 2025-Q4/
│           ├── 2025-12-03-voucher-expiry-field.yaml
│           └── ...
├── cckn/
│   ├── postgres-query-planner.md
│   ├── jwt-verifier-multi-kid.md
│   └── ...
├── completion-reports/
│   ├── 2026-04-22-jwt-rotation.md
│   └── ...
├── escalation-log.md                 # role: escalation log — append-only
└── supervision-log.md                # role: supervision log — append-only
```

**Notes:**

- Retiring a manifest is `mv manifests/active/<id>.yaml manifests/retired/<quarter>/<id>.yaml` plus `status: retired` flip in the same commit. Doing one without the other creates a SoT-vs-filesystem contradiction.
- `index.md` is regenerated by a small script that walks the directory; humans should not hand-edit it.
- CCKN files are referenced from manifests via relative path; older-than-12-months CCKNs require re-verification per [`docs/cross-change-knowledge.md`](../cross-change-knowledge.md).

## Pattern 2 — Git LFS-friendly variant

**When to use:** evidence artifacts (binary fixtures, large logs, screenshot bundles, profiler traces) are large enough that committing them under the same prefix as text-shaped manifests bloats the repo.

```
.ai-memory/                           # Git-tracked, plain-text only
├── index.md
├── manifests/
│   ├── active/...
│   └── retired/...
├── cckn/...
├── completion-reports/...
├── escalation-log.md
├── supervision-log.md
└── evidence-pointers/                # role: evidence index
    └── <change_id>/
        └── pointers.yaml             # maps evidence type → LFS path

.ai-memory-evidence/                  # Git LFS or a separate object store
├── 2026-04-26-search-p95-mission/
│   ├── load-test-2026-04-26.html
│   ├── golden-parity-2026-04-26.json
│   └── ...
└── ...
```

**Notes:**

- `evidence-pointers/<change_id>/pointers.yaml` is the bridge between manifests (which cite `artifact_location: ".ai-memory-evidence/<change_id>/<file>"`) and the LFS-stored artifacts. It exists so a session reading a manifest can resolve evidence without pulling LFS.
- The split is purely about storage cost. The manifest's `evidence_plan[*].artifact_location` still resolves; it just resolves to an LFS path that the runtime fetches on demand.
- This pattern still satisfies the four runtime requirements: writable (manifests + LFS), retrievable (LFS pull on demand), versioned (LFS commits), traceable (`artifact_location` + LFS commit hash).

## Pattern 3 — Monorepo federated variant

**When to use:** monorepo with several independently-released packages where each team wants its own project-memory but a session at the root needs a unified view.

```
.ai-memory/
└── index.md                          # federated index; aggregates per-package
                                      # role: index — points at each package's local memory

services/
├── search-api/
│   └── .ai-memory/                   # local to this package
│       ├── manifests/active/...
│       ├── cckn/...
│       └── ...
├── identity/
│   └── .ai-memory/...
└── billing/
    └── .ai-memory/...

apps/
├── web/
│   └── .ai-memory/...
└── mobile/
    └── .ai-memory/...
```

**Notes:**

- Top-level `.ai-memory/index.md` is regenerated by a script that walks each package's local `.ai-memory/` and aggregates a unified view. Sessions starting at repo root read this; sessions working inside a package read the local memory.
- Cross-package changes use `part_of` (manifest references the parent change-id at the umbrella scope) — see [`docs/change-decomposition.md`](../change-decomposition.md). The umbrella manifest may live at the root `.ai-memory/`; child manifests live in their respective packages.
- A package whose changes never cross package boundaries can have its `.ai-memory/` be largely self-contained. Cross-package consumers are still listed in `sot_map[*].consumers` per [`docs/source-of-truth-patterns.md`](../source-of-truth-patterns.md).

## What none of these patterns specify

- **Storage backend.** All three examples assume a Git-tracked filesystem. A runtime that exposes project memory through MCP tool calls, an HTTP API, or a vector DB is equally conformant if it provides the role split and the four runtime requirements.
- **Access protocol.** A bridge may surface project memory as `read_file` calls, `query_memory(role=active_manifest, since=2026-04-01)`, or any other interface — the layout describes the *content shape*, not the read path.
- **Naming.** `.ai-memory/`, `.agent-context/`, `memory/`, `.foo-state/` — pick whatever fits the repo's conventions. The role table in [`docs/ai-project-memory.md`](../ai-project-memory.md) §Recommended on-disk layout is what is pinned; the directory name is not.

---

## Anti-patterns specific to layout

- **Status flip without filesystem move (Pattern 1 / 3 only).** Manifest `status: retired` but the file still lives under `manifests/active/` — a session listing active manifests sees a retired one and acts on it.
- **Index hand-edited.** Humans curating the index introduces drift between index and ground truth. Regenerate it.
- **Evidence inlined under manifests/ when LFS would fit (Pattern 1 misuse).** Bloats the repo and makes manifests slow to clone for sessions that only need the text.
- **Per-package memory with no federated index (Pattern 3 misuse).** A session starting at the repo root cannot answer "what is in flight across the monorepo?" without walking every package — the federated index exists to make that question one-file-cheap.
- **Layout-by-stack-bridge instead of layout-by-role.** A bridge that pins directory names per stack ("Flutter projects must use `.flutter-memory/`") fragments the ecosystem and gains nothing — the role split is what governance cares about.
