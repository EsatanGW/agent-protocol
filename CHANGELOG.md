Prior state: CCKN's **write** side was prescribed ("*the Planner creates or extends a CCKN as part of the Phase 1 Investigate re-entry*" / "*the Planner evaluates during Phase 1 Investigate whether the change's learning content belongs in a CCKN*") but the **query** side was only passively mentioned ("*Any manifest field that benefits from pre-existing knowledge cites the CCKN by path*") — no named procedural step said "query first, before investigation proper begins." ## [1.13.1] - 2026-04-22

### Added

- **CI validator for Pattern C cluster disjointness (`.github/scripts/validate-cluster-disjointness.py`)** — closes the "validator-enforced" integrity gap that 1.13.0 shipped with: `cluster-parallelism.md`, the schema description on `implementation_clusters`, and the 1.13.0 ROADMAP Phase log all promised a validator that did not exist. The new script enforces the invariant: given a manifest with `implementation_clusters`, expand each cluster's `scope_files` globs against the repo root, compute pair-wise set intersections; any non-empty intersection is a disjointness violation and fails the run. Exit codes match the existing validator convention (0 OK / 1 violation / 2 tool error). Covers three invocation shapes: (a) `--self-test-only` — runs 5 embedded synthetic fixtures in a tmpdir, always passes regardless of repo content; (b) default — runs self-test then scans `templates/*.yaml` and `examples/starter-repo/**/change-manifest.*.yaml` for any with `implementation_clusters` filled; (c) `--manifest <path>` — ad-hoc single-file check. The 5 self-test cases exercise: disjoint by top-level dirs, disjoint by subdir, subsuming pattern (`db/**` vs `db/migrations/*.sql`) → violation, identical patterns → violation, empty `implementation_clusters` → no-op.
- **`cluster-disjointness` CI job (`.github/workflows/validate.yml`)** — new job that runs the new script on every push to main and every pull request. Joins the existing `schema-syntax` / `schema-drift` / `changelog-drift` / `template-conformance` / `hooks-selftest` / `validator-python-tests` / `validator-node-tests` / `version-consistency` jobs. Matches the existing no-untrusted-input property of this workflow (only hardcoded `pip install` + `python3 .github/scripts/...` commands; no `github.event.*` interpolation).

## [1.13.0] - 2026-04-22

### Added

- **Cluster parallelism reference (`skills/engineering-workflow/references/cluster-parallelism.md`)** — new normative reference defining **Pattern C — Phase 4 cluster-parallel canonical Implementers**, the within-Phase-4 complement to 1.11.0's Pattern-A/B non-canonical sub-agent fan-out and 1.12.0's between-phase overlap zones. Core mechanism: the Planner declares 2–4 file-disjoint implementation clusters in `implementation_clusters` at Phase 2 approval; at Phase 4 the Planner spawns one canonical Implementer per cluster in a single tool-call batch (cache-window rule). Each spawned Implementer has the full canonical Implementer envelope (read, write, shell) and writes its own cluster's code and evidence directly — this is the key distinction from Patterns A/B (non-canonical sub-agents that return findings for the canonical role to synthesize). A single Reviewer downstream performs a **cross-cluster cross-cutting gap check**: issues that emerge only at cluster intersections (e.g. cluster A's API field matches cluster B's frontend call but cluster C's analytics event still references the old name) that no individual Implementer would catch. Four mandatory invariants: (1) file-disjoint scope per cluster (validator-enforced pair-wise disjointness); (2) single-batch spawn; (3) any cluster's Discovery-loop trigger halts all clusters by default (conservative — other clusters may be building on an invalidated plan); (4) Reviewer performs cross-cluster cross-cutting gap check. Cluster cap 2–4 (symmetric with Patterns A/B visibility argument). Full mode only; Lean and Zero-ceremony explicitly excluded. Anti-patterns documented: "mostly" disjoint clusters, serial spawn, one-cluster-Discovery-ignored-while-others-continue, per-cluster isolated Review without cross-cluster check, Implementer writes outside scope_files, cluster identity shared between clusters, Pattern C used in Lean mode, heterogeneous-envelope-per-cluster.

## [1.12.0] - 2026-04-22

### Added

- **Phase overlap zones reference (`skills/engineering-workflow/references/phase-overlap-zones.md`)** — new normative reference defining **between-phase pipeline parallelism**, the complement to 1.11.0's within-phase fan-out. Core concept: specific prep work for Phase N+1 may begin in session-scoped working space before Phase N's gate passes, provided three properties hold: (1) the prep does not depend on Phase N's gate output, (2) it produces an artifact the downstream phase will consume, (3) it is **discarded and redone** if Phase N's gate fails. Five named zones: **P1 → P2** (change-map skeleton from stable surface + SoT list), **P2 → P3** (test-plan skeleton from stable ACs), **P3 → P4** (baseline verification environment — harness scaffold, CI branch, screenshot baseline, metric snapshot — explicitly not implementation code), **P4 → P5** (Reviewer context-pack pre-distillation + reference-sampler seeding; approval language explicitly forbidden during overlap), **P5 → P6** (pre-filter structural scan + sign-off template pre-fill of AC headers; verdict language explicitly forbidden). **Three zones explicitly NOT overlap zones**: P0 → P1 (classification before clarification fails too often), P6 → P7 (pre-filled completion report is an anti-rationalization hazard), P7 → P8 (observation waits for real post-delivery signal). Five mandatory constraints: (1) no canonical-role work-ahead, (2) no manifest-field writes in overlap, (3) discard-on-fail recorded in `Phase log`, (4) ≤20% of downstream phase's work (past 20% the prep has become execution and is bypassing the gate by degree), (5) Full mode only. Relation to parallelization-patterns: orthogonal — fan-out is within-phase sub-agent parallelism; overlap is between-phase pipeline parallelism; the two stack in the same initiative without interference.
- **Glossary entry — Phase overlap zone (`docs/glossary.md`)** — new canonical term in the Process section, sibling to Fan-out / Fan-in / Context pack / Parallel group. Cross-references the five named zones, the three excluded boundaries, and the canonical reference. Full-mode-only constraint and ≤20% soft cap surfaced at the glossary level so the distinction between within-phase fan-out and between-phase overlap is discoverable from the vocabulary layer without reading the full reference.
- **Phase-file overlap cues (`skills/engineering-workflow/phases/phase{1,2,3,4,5}-*.md`)** — each phase file gains a terse one-paragraph "Optional: P{N} → P{N+1} overlap zone" section pointing at the relevant zone in the new reference. Phase 4's cue explicitly names approval-language-during-overlap as forbidden (anti-rationalization hazard); Phase 5's cue explicitly names verdict-language-during-overlap as forbidden (same hazard, P5 → P6 boundary). Phase files remain the 20–50-line size-style the skill uses.

## [1.11.0] - 2026-04-22

### Added

- **Parallelization patterns reference (`skills/engineering-workflow/references/parallelization-patterns.md`)** — new normative execution-layer reference defining two canonical parallel sub-agent fan-out patterns: **Pattern A — Phase 1 surface-parallel investigators** (Planner-owned; 2–4 read-only investigators per surface, single-batch spawn, Planner performs fan-in synthesis), and **Pattern B — Phase 5 specialized audit fan-out** (Reviewer-owned; security / cross-cutting / evidence-reference / acceptance-coverage sub-agents inheriting the Reviewer's read-only envelope, Reviewer performs fan-in synthesis). Five mandatory disciplines lifted out of principle 9 to the sub-agent layer: (1) **cache-window rule** — fan-out sub-agents are spawned in a single tool-call batch so the prompt-cache prefix hits; serial spawning erases the parallelization benefit; (2) **context pack rule** — every fan-out shares a pre-distilled context artifact so sub-agents do not redundantly re-read `docs/`; (3) **fan-in discipline** — synthesis is performed by the canonical role itself, never delegated, with every sub-agent return read in full; (4) **cross-cutting gap check** — the specific failure mode parallelization introduces is a gap visible only in the intersection of two sub-agents' findings; the check asks for union-emergent gaps, contradictions between sub-agents, and suspiciously empty returns; (5) **audit trail** — the fan-out is recorded in the manifest's new `parallel_groups` field. Full-mode only; Lean / Zero-ceremony explicitly excluded by the existing role-composition contract. Phase applicability table included: Phase 1 → Pattern A; Phase 5 → Pattern B; Phase 4 intentionally has no parallel pattern (Pattern 3 covers serial Implementer + test-writer composition).
- **Context pack reference and template (`skills/engineering-workflow/references/context-pack.md` + `skills/engineering-workflow/templates/context-pack-template.md`)** — new session-scoped working-space artifact shared across every sub-agent in a fan-out group, so sub-agents do not each re-read the full methodology tree. Purpose: keep fan-out economics net-positive and increase sub-agent consistency (shared SoT candidates, shared surface list, shared terminology). Six required contents: change identity, pre-classified SoT candidates, surfaces in scope with sub-agent scope boundaries, terminology snapshot, hard boundaries, return-slot shape. Four exclusions: full document copies, the Change Manifest itself, implementation diffs, cross-role context. Budget: under 400 words typically; hard cap 800 (symmetric with the handoff-prompt budget). Lives in working space per Rule 5a; never persisted, never cited as evidence.
- **Patterns 5 and 6 in `reference-implementations/roles/role-composition-patterns.md`** — non-normative companion to the new normative parallelization doc. Pattern 5 (Planner + surface-parallel investigators) and Pattern 6 (Reviewer + specialized audit fan-out) extend the existing four serial patterns with the parallel variant. The existing "three canonical roles remain canonical" invariant is unchanged; fan-out sub-agents are still non-canonical, still have distinct identities, still inherit envelope, still do not write manifest fields. TL;DR and "What this document is not" sections updated from "four patterns" → "six patterns"; anti-pattern table gains three parallel-specific entries (serial spawn of fan-out, missing `parallel_groups` entry, missing cross-cutting gap check).
- **`parallel_groups` Change Manifest field (`schemas/change-manifest.schema.yaml` + `schemas/change-manifest.schema.json` + `docs/change-manifest-spec.md`)** — new optional top-level array recording every parallel fan-out that occurred within the change. Each entry captures: `owning_role` + `owning_identity` (canonical role that fanned out), `phase`, `pattern` (enum `A_surface_investigators` / `B_specialized_audit` / `other`), `context_pack_summary`, `sub_agents` (`minItems: 2` / `maxItems: 4` — a 1-wide fan-out is a serial sub-agent call, a >4-wide fan-out degrades cross-cutting visibility), and `synthesis` (with `performed_by_identity` that MUST equal `owning_identity`, `manifest_fields_written`, and `cross_cutting_gap_check` recording gaps / contradictions / empty returns). Backward-compatible: pre-1.11 manifests and Full-mode changes that did not fan out both validate without this field. ## [1.0.0] - 2026-04-19

Initial public release. Tool-agnostic engineering workflow plugin for AI coding agents, usable across Claude Code, Cursor, Gemini CLI, Windsurf, Codex, Aider, OpenCode, or any AI runtime that reads `AGENTS.md`.

### Included

- **Operating contract** — `AGENTS.md` defines the universal runtime contract: honest reporting, scope discipline, source-of-truth rules, surface-first analysis, evidence requirements, Change Manifest contract, phase-gate discipline.
- **Engineering workflow skill** — `skills/engineering-workflow/` provides Lean and Full modes, phase minimums for Phase 0 through Phase 8, capability-category guidance, and spec / plan / test / completion templates. Supporting references cover the mode decision tree, discovery loop, resumption protocol, startup checklist, and misuse signals.
- **Methodology documentation** — `docs/` contains the four canonical surfaces plus nine extension surfaces (asset, experience, performance-budget, data-quality, compliance, hardware-interface, real-world, model, uncontrollable-external), ten Source-of-Truth patterns, the L0–L4 breaking-change severity matrix, three rollback modes (Reversible / Forward-fix / Compensation-only), cross-cutting concerns, the AI operating contract, security and supply-chain disciplines, change decomposition, team / org-scale disciplines, AI project memory, multi-agent handoff, automation contract, and phase-gate discipline.
- **Change Manifest contract** — `schemas/change-manifest.schema.yaml` plus four worked example manifests (CRUD, mobile offline, game gacha, multi-agent handoff progression). The schema enforces human-approver waivers, expiry timestamps, bidirectional decomposition links, and phase-appropriate narrative fields.
- **Stack bridges** — `docs/bridges/` maps the tool-agnostic methodology to concrete stacks: Flutter, Android Kotlin + XML, Ktor, Unity 3D. Bridge files are the only place where specific vendor / framework / language names appear. The stack-bridge template (`docs/stack-bridge-template.md`) supports adding your own stack.
- **Automation contract** — capability-level specification (`docs/automation-contract.md`) plus normative algorithm (`docs/automation-contract-algorithm.md`) defining three forced check layers (structural / cross-reference / drift), exit-code stability, and offline operability. A non-normative reference validator (`reference-implementations/validator-posix-shell/`) built on POSIX shell plus `yq` plus a pluggable schema validator illustrates the contract.
- **Worked examples** — `docs/examples/` covers ten domains: worked (batch-resend invoices), bugfix (pagination-filter desync), refactor (order-status canonical alignment), migration-rollout (structured-preferences SoT transition), game-dev (in-game shop), game-liveops (three-track asset/config/binary strategy), mobile-offline-feature, ml-model-training, data-pipeline, embedded-firmware (OTA with hardware-interface + real-world surfaces).
- **Multi-agent entry points** — packaged simultaneously as a Claude Code plugin (`.claude-plugin/`), a Cursor rules bundle (`.cursor/rules/`), a Gemini CLI instruction set (`GEMINI.md`), Windsurf rules (`.windsurfrules`), and an AGENTS.md-compatible contract for Codex / Aider / OpenCode.

### Design invariants

- **English-only.** All normative content — methodology, skills, schemas, templates, operating contract, READMEs — is authored in English.
- **Tool-agnostic.** Specific vendor, model, framework, or product names appear only in `docs/bridges/` and `reference-implementations/`, both of which are explicitly marked non-normative. Methodology docs, the skill layer, the schema, and the Change Manifest templates remain stack-neutral.
- **Capability categories, not tool names.** Every operational instruction names a category (file read, file write, code search, shell execution, sub-agent delegation) rather than a specific tool, so the plugin ports cleanly between AI runtimes.
- **Canonical terminology held fixed.** Severity L0–L4 (Additive / Behavioral / Structural / Removal / Semantic-reversal), rollback modes 1/2/3 (Reversible / Forward-fix / Compensation-only), four canonical surfaces plus extension surfaces, SoT patterns addressed by number, automation tiers L0–L3, three multi-agent roles by responsibility (Planner / Implementer / Reviewer).
