## [1.23.0] - 2026-04-27

Cluster B of the 1.8.0-SDD-import reverse audit (see `working/2026-04-27-1-8-0-sdd-audit.md`). Two normative changes in `docs/phase-gate-discipline.md`: **Rule 5a Restrict** — Working Space Convention no longer fires at every Full-mode phase boundary; it now fires when (a) `breaking_change.level` ≥ L2, (b) the boundary is Phase 5 (Reviewer audit), or (c) the phase produces intermediate artifacts that another invocation must read (Pattern A/B/C fan-out, long-running-delegation D2 progress, phase overlap-zone prep). **Rule 6 Merge** — phase re-entry decision table re-organized from seven source variations into four destination-phase rows; original variations preserved verbatim as `(a)`–`(g)` sub-cells. Forced-Full per `CLAUDE.md §5` (both edits change normative claims in a canonical SoT file).

User-facing impact: a Full-mode L0 / L1 documentation edit (the dominant work shape for this repo's recent canonical edits) no longer requires a buffered working-space file at every phase boundary. Estimated working-space file reduction ~60% on low-severity Full work; high-severity changes, Phase 5 audit, and parallel patterns retain the discipline because that is where its audit value compounds.

### Changed

- **`docs/phase-gate-discipline.md` Rule 5a — Working Space Convention** (**L1**) — both the master ceremony-scaling table at line 33 and the Rule 5a body's ceremony-scaling subsection are updated. The Full column changes from "Applies at every phase boundary" to a three-trigger formulation: (a) `breaking_change.level` ≥ L2; (b) Phase 5 (Reviewer audit always uses a working space — Anti-Rationalization Rule 1's audit-against-working-space requires it); (c) the phase produces intermediate artifacts another invocation or session must read before the canonical artifact lands — covering Pattern A/B/C fan-out (`role-composition-patterns.md`), long-running-delegation D2 progress (`long-running-delegation.md`), and phase overlap-zone prep (`phase-overlap-zones.md`). None of (a)/(b)/(c) → direct-write to canonical permitted. Trigger (c) is named explicitly so the three load-bearing consumers do not need defensive re-statement; their existing pointers (`glossary.md` Context-pack / Phase overlap zone definitions; `long-running-delegation.md` D2 working-space pin; `phase-overlap-zones.md` overlap-prep working-space pin; `parallelization-patterns.md` context-pack working-space line; `context-pack.md` working-space requirement) remain valid without edit. ### Why minor, not patch

Both edits change normative content in a canonical SoT file (`docs/phase-gate-discipline.md`). The Rule 5a Restrict narrows the trigger condition (an existing rule's firing surface shrinks) — that is an **L1 normative-claim change**, not a wording clarification. The Rule 6 table re-organization re-shapes a normative table even though the seven source variations are preserved 1-for-1; readers who hardcoded "row 3 = breaking-change rises" rather than the underlying variation will need to re-orient. Per `mode-decision-tree.md §Scenarios that force Full → Canonical methodology content edit (L1+)`, both are forced-Full triggers requiring a minor bump.

The bundling matches `docs/cross-change-knowledge.md`'s recommendation for L1+ work that touches a single SoT file in one pass: pay the cross-change cognitive cost once per release rather than spreading two L1+ edits in `phase-gate-discipline.md` across two minor releases.

### Migration notes

- **External consumers citing Rule 5a's old "every phase boundary" wording** — re-orient to the three-trigger formulation. The dominant case (Full-mode L0 / L1 single-surface edits) now permits direct-write; high-severity / Phase-5 / parallel-pattern work retains the discipline.
- **External consumers citing Rule 6 by row number** (`row 3`, `row 5`, …) — switch to citing by source variation label `(a)`–`(g)` (preserved verbatim) or by destination phase (`Phase 0`, `Phase 1`, `Phase 2`, `Phase 4`). The seven source variations are the stable identifiers across the 1.8.0 → 1.23.0 form change.
- **External consumers using `re-entry-trigger/` module** — no API change; `suggest_reentry()` returns the same five phase-enum strings. README and DEVIATIONS now describe the implementation in terms of the 1.23.0 doc form, but the call surface is unchanged.

Rule 6 source variations are cited by their existing labels, not new identifiers. No schema changes (Rule 5a and Rule 6 are normative-prose rules, not schema-validated fields). ## [1.21.0] - 2026-04-27

This release adds two new universal AI-agent disciplines — **Agent persona discipline** (§9) and **Output craft discipline** (§10) — to `AGENTS.md §Core operating contract`, expanding the operating contract from 8 rules to 10. Forced-Full per `mode-decision-tree.md §Canonical methodology content edit (L1+)` (adding new normative rules to the SoT operating contract).

### Added — canonical content

- **`AGENTS.md §9 — Agent persona discipline`** (new operating-contract rule). Every AI-agent invocation reasons from a real domain-expert persona (system architect / motion designer / UX designer / deck designer / prototyper / …), not from the generic "I am an AI assistant" default. Persona is **selected by the medium of the output**, not by the format of the input, and shifts when the medium shifts. Persona is **orthogonal** to the canonical workflow role (§7) and to specialist sub-agent roles; role names the workflow position and tool envelope, persona names the domain stance the agent reasons from. Persona never overrides tool permissions, anti-collusion, phase-gate discipline, evidence requirements, or non-fabrication rules. Persona-as-permission-escalation ("as a senior architect I can approve this myself") is explicitly forbidden. Full SoT: `docs/agent-persona-discipline.md`.
- **`AGENTS.md §10 — Output craft discipline`** (new operating-contract rule). Every output the agent produces — code, prose, manifests, summaries, UI, deliverables — must clear three rules: (1) **every element earns its place** ("what would be lost if this were removed?" — if "nothing, but it fills the space", remove it; default to omission, treat empty space as content), (2) **output adapts to its medium; the AI default is rejected** (default styling is a fingerprint not a feature; the persona declared per §9 picks the styling vocabulary; web tropes, dummy data, emoji-as-icons, ornament SVG, layout-balancing sections are filler everywhere they are not part of the medium), (3) **summaries are caveats and next steps, not recap** (the diff and the manifest are the record; the agent's contribution at the summary slot is what those records do not say). Rules are stated as principles, not as exhaustive trope lists, so they remain medium-agnostic as new AI defaults emerge. Full SoT: `docs/output-craft-discipline.md`.
- **`docs/agent-persona-discipline.md`** — new Tier-2 canonical SoT for §9. Open-vocabulary persona axis (illustrative entries: motion designer, UX designer, deck designer, prototyper, system architect; projects may register their own). Six anti-patterns: persona-less reasoning, frozen persona, multi-persona stacking, persona-as-permission-escalation, persona-medium mismatch, decorative persona.
- **`docs/output-craft-discipline.md`** — new Tier-2 canonical SoT for §10. Three rules with self-check trio at delivery time. Cross-references `agent-persona-discipline.md` (persona picks the styling vocabulary), `ai-operating-contract.md §7` (Rule 3 is the summary-slot application of "Difference > repetition"), and `ai-operating-contract.md §1 / §9` (inventing data violates non-fabrication too).
- **`docs/examples/agent-persona-example.md`** — new worked example. Same prompt under three personas (UX designer / motion designer / system architect) producing categorically different work; mid-session persona switch (architect → deck designer when the medium shifts to a slide deck); persona-as-permission-escalation anti-pattern.
- **`skills/engineering-workflow/templates/manifests/change-manifest.example-methodology-evolution.yaml`** — new Change Manifest for *this release itself*, doubling as Example 7 in the manifest examples set. Schema-validated (117 lines) record covering only the schema-required fields (surfaces / sot_map / breaking_change / rollback / evidence_plan) plus the teaching content (review_notes documenting the structural review gap; approvals recording the SoT-author authorization; handoff_narrative pointing back at this CHANGELOG). `cross_cutting`, `implementation_notes`, and `residual_risks` are deliberately omitted — they are either trivial (all "none" / "low") or duplicated from CHANGELOG; the manifest's manifest-only abbreviated ceremony deliberately avoids retroactive ceremony for fields whose value would be theatre. Honestly documents the canonical Phase 5 review gap: rather than fabricate self-affirming `pass` entries (which would violate `multi-agent-handoff.md §Anti-rationalization Rule 3`), `review_notes` marks the canonical-review-applicability finding as `needs_human_decision`, describes the structural circularity (SoT author == Implementer; no distinct Reviewer identity available that is not bound by the contract being changed), and lists what *was* verified mechanically (cross-reference resolution, bridge sync, rule-numbering preservation, schema validation). Approval scope explicitly authorizes the deviation pattern as a precedent for future methodology self-evolution.

### Added — supporting sub-sections

- **`docs/ai-operating-contract.md` §1 — Information sufficiency**. Symmetric rule to §1 honest-uncertainty: when info is sufficient, **act**; asking to re-confirm what is determinable is itself a misuse. Rule is symmetric — insufficient → ask explicitly; sufficient → act and report; the silent middle path of "act but pretend confirmed" remains forbidden.
- **`docs/ai-operating-contract.md` §6 — Finish-and-verify sequence (compressed three-step view)**. Navigation aid for the three-step terminal pattern: (1) call done = run the role-specific pre-handoff self-check, (2) if it fails, fix and call done again under the same identity, (3) when it passes, hand to a different-identity verifier. Not a new normative claim — every step references existing canonical contracts (Pre-handoff self-check, Discovery loop, anti-collusion rule); naming the three-step structure prevents "declare done, fix in same turn, never cross an external gate."
- **`docs/ai-operating-contract.md` §7 — Summary format**. Explicit format for the conversational summary slot, applying Rule 3 of §10: caveats + next steps only, lead with the load-bearing fact, no recap, no closing flourish.
- **`docs/glossary.md` §Six-stage compact view (alias)**. Six-stage view (understand → explore → plan → build → verify → minimal summary) documented as a compact alias of Lean's six steps and Full's nine phases. Explicit non-claim: not a third execution mode.

### Added

- **`reference-implementations/roles/role-composition-patterns.md` §Shape of a composition → §Invocation lifecycle** — new subsection closing a previously-undocumented mental-model gap: sub-agent calls are **one-shot invocations**, not long-lived processes. The section names the invocation-lifecycle rule (spawn → run → return → terminate), introduces the **invocation handle** concept (some runtimes expose a handle that can seed a new invocation inheriting prior memory), and distinguishes **continuation primitive / state reconstruction** from **state continuation**. - **`reference-implementations/roles/role-composition-patterns.md` §Anti-patterns — two new rows** covering the concrete runtime-bridge failure modes the new §Invocation lifecycle rule forbids: (1) canonical role attempts cleanup / termination on a sub-agent whose invocation has already returned (wastes tool calls; on runtimes that return a "not found" error, the error further pollutes the canonical role's mental model with a false signal); (2) canonical role treats a stale invocation handle as identity-equivalent to a still-running sub-agent (e.g. reuses the handle for audit work) — a new invocation seeded from a handle is a new identity whose anti-collusion constraints (Rule 5 / Pattern 6) must be re-evaluated, not inherited.

Backward-compatible: pre-1.17 changes, compositions that did not misuse handles, and runtimes that do not expose any handle primitive at all remain fully valid.

### Why this release closes a mental-model gap

§Shape of a composition (1.12.0) defined the four parts of a composition — canonical invocation / sub-agent invocation / consolidation / handoff — in a way that is correct but silent on invocation lifecycle. The composition contract says the canonical role reads the sub-agent's return and continues; it does not say the sub-agent invocation has terminated at that point. For runtimes where a sub-agent is literally a long-lived process this silence is harmless. For runtimes that expose an **invocation handle** for continuation (the increasingly common pattern), the silence was load-bearing in the wrong direction: a canonical role could plausibly read the handle as "my sub-agent is still running, I should stop it now," which is both a wasted tool call and — if the same handle is later re-used for audit work — a structural anti-collusion violation per Rule 5 / Pattern 6. This release names the lifecycle rule in the canonical reference, records the two concrete failure modes as anti-patterns, and surfaces the concept at the vocabulary layer so readers do not need to infer it from the composition shape alone.

Specific runtime bridges (e.g. `agents/*.md`, `.cursor/rules/*.mdc`, `reference-implementations/roles/*.md`) may later translate these concepts into concrete tool names from their runtime, but the normative layer named in this release stays tool-agnostic.
