# Core Principles

1. Manage change, not just code
2. Find source of truth before patching consumers
3. Mark affected surfaces before choosing a workflow mode
4. Pick the lightest workflow that still preserves quality
5. Require verification and evidence before completion
6. Increase documentation and handoff rigor when public behavior changes
7. Close every phase with an explicit, named gate; track multi-phase initiatives in the repo's `ROADMAP.md`; commit at every passed gate when version control is available
8. Read any user-provided spec document in full before planning; re-reference the spec at every subsequent phase; treat discovered spec conflicts as escalation, not silent deviation
